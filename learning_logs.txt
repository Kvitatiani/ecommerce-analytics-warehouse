=== SQL — DDL ===

CREATE SCHEMA IF NOT EXISTS bronze;
  ^ IF NOT EXISTS = idempotent (safe to run multiple times)

CREATE TABLE IF NOT EXISTS bronze.products_raw (
    id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    products_payload JSONB,
    ingestion_timestamp TIMESTAMP DEFAULT NOW(),
    source TEXT,
    batch_id TEXT
);
  ^ GENERATED ALWAYS AS IDENTITY = modern auto-increment (replaces SERIAL)
  ^ DEFAULT NOW() = auto-fills timestamp if not provided
  ^ JSONB = binary JSON, faster queries, supports indexing

TRUNCATE bronze.categories_raw;   -- fast delete, resets IDs
DELETE FROM bronze.categories_raw; -- slower, doesn't reset IDs
TRUNCATE silver.categories CASCADE; -- truncates + anything referencing it (foreign keys)

=== JSONB QUERYING ===

SELECT payload->>'title' FROM bronze.products_raw;   -- extracts as TEXT
SELECT payload->'category' FROM bronze.products_raw;  -- keeps as JSON object
SELECT payload->'category'->>'id' FROM bronze.products_raw; -- nested: get JSON object, then extract field as text

=== TYPE CASTING ===

(payload->>'id')::INTEGER           -- cast text to integer
(payload->>'price')::NUMERIC(10,2)  -- cast to decimal with precision
(payload->>'creationAt')::TIMESTAMP -- cast to timestamp
  ^ parentheses matter! Wrap the whole extraction before casting
  ^ WRONG: payload->>'creationAt'::TIMESTAMP  (casts the key name, not the value)
  ^ RIGHT: (payload->>'creationAt')::TIMESTAMP

=== UPSERT (INSERT ... ON CONFLICT) ===

INSERT INTO silver.categories (category_id, category_name)
VALUES (1, 'Clothes')
ON CONFLICT (category_id) DO UPDATE SET
    category_name = EXCLUDED.category_name;
  ^ EXCLUDED = the row that was trying to be inserted
  ^ requires a UNIQUE or PRIMARY KEY constraint on the conflict column
  ^ makes transformations idempotent — safe to re-run

=== WINDOW FUNCTIONS — DEDUPLICATION ===

WITH ranked AS (
    SELECT *,
        ROW_NUMBER() OVER (PARTITION BY email ORDER BY id) AS rn
    FROM table
)
SELECT * FROM ranked WHERE rn = 1;
  ^ PARTITION BY = group rows (like GROUP BY but keeps all rows)
  ^ ORDER BY = determines which row gets rn=1 (lowest id = first)
  ^ WHERE rn = 1 = keep only one row per group
  ^ Used to deduplicate: e.g., 30 rows with same email → keep the one with lowest id

=== REGEX FILTERING IN POSTGRESQL ===

column ~ 'pattern'    -- matches regex
column !~ 'pattern'   -- does NOT match regex

-- Only letters, spaces, hyphens:
name ~ '^[A-Za-z\s\-]+$'

-- Valid email pattern:
email ~ '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$'

=== FOREIGN KEYS ===

product_category_id INTEGER REFERENCES silver.categories(category_id)
  ^ ensures value must exist in referenced table
  ^ order matters: referenced table must be created/populated first
  ^ TRUNCATE on referenced table needs CASCADE or truncate dependent table first

-- Validate during insert:
AND (payload->'category'->>'id')::INTEGER IN (SELECT category_id FROM silver.categories)

=== PYTHON — BRONZE LOADER ===

Key pattern for inserting JSONB:
  from psycopg2.extras import Json
  cursor.execute(
      "INSERT INTO table (payload, source, batch_id) VALUES (%s, %s, %s)",
      (Json(row), source_name, batch_id)
  )
  ^ Json() wraps python dict for JSONB column
  ^ %s placeholders prevent SQL injection (never use f-strings for values)
  ^ connection.commit() required or data is lost

Config-driven ingestion with list of dicts:
  table_config = [
      {"table_name": "bronze.categories_raw", "endpoint": "categories", "payload_column": "categories_payload"},
      ...
  ]
  for table in table_config:
      data = get_api_data(table["endpoint"])
      if data:
          insert_data(data, table["table_name"], ...)

Full refresh pattern (truncate + reload):
  ^ TRUNCATE table before insert — works when API returns full dataset each time
  ^ Only truncate AFTER confirming API returned data (check if data is not None/empty)

=== CONCEPTS LEARNED ===

- Bronze layer = raw, untouched API data as JSONB. Append-only, never update/delete.
- Silver layer = cleaned, typed, deduplicated data. One row per entity. Enforces constraints.
- Idempotency = safe to run multiple times, same result (IF NOT EXISTS, UPSERT, TRUNCATE+reload)
- Batch ID = shared timestamp across all rows in one pipeline run
- ingestion_log = tracks every run (success/failure, record count, errors)
- .env for credentials, .env.example committed as template, .env in .gitignore
- Full refresh = truncate and reload entire table. Good when source gives full dataset.
- Referential integrity = foreign keys + query-level checks ensure data consistency across tables
- Transformation order matters: categories → products (because of foreign key dependency)
- Data quality filtering: regex, NOT NULL checks, deduplication — all happen in silver, not bronze
- Public APIs have junk data — silver layer is where you clean it
